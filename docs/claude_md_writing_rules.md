# CLAUDE.md 작성 규칙

## 목적
이 문서는 저장소의 `CLAUDE.md` 파일들을 어떻게 작성할지 정의한다.
루트 `CLAUDE.md`는 프로젝트 전체를 처음 읽는 에이전트의 진입점이고, 하위 폴더의 `CLAUDE.md`는 해당 폴더에서 실수하지 않게 만드는 작업 지침이다.

## 적용 범위
- 루트 파일: `CLAUDE.md`
- 하위 폴더 파일: `dashboard/CLAUDE.md`, `chatgpt-usage-tracker/CLAUDE.md`, `supabase/CLAUDE.md`, `docs/CLAUDE.md`, `docs/HARNESS/CLAUDE.md`, `scripts/CLAUDE.md`
- 루트 문서는 공통 불변조건, 전체 지도, 기본 작업 흐름, 공통 도구를 담는다.
- 하위 문서는 해당 폴더의 구조, 금지 사항, 작업 순서, 검증 방법을 담는다.

## 루트 CLAUDE.md 권장 섹션
```md
## 제약 조건
## 프로젝트 소개
## 디렉터리 지도
## 작업 흐름
## 도구
## 아키텍처
## 도메인 컨텍스트
## 개발 원칙
## 코딩 컨벤션
```

## 하위 CLAUDE.md 권장 섹션
하위 폴더는 루트와 같은 섹션을 반드시 모두 가질 필요는 없다.
다만 아래 섹션을 기본값으로 삼고, 폴더 성격에 맞지 않는 섹션은 줄이거나 생략한다.

```md
## 제약 조건
## 이 폴더의 역할
## 디렉터리 지도
## 작업 흐름
## 도구
## 도메인 컨텍스트
## 폴더별 규칙
```

하위 문서에는 `개발 원칙`을 반복하지 않는다.
루트의 개발 원칙을 따르고, 하위 문서에는 그 폴더에서 필요한 구체적 규칙만 둔다.

## 제약 조건 작성 규칙
제약 조건에는 반드시 지켜야 하는 불변조건만 넣는다.

좋은 제약 조건:
- 메시지 본문, 프롬프트, 응답 전문 저장/표시/로그 출력 금지
- 비밀값/로컬 설정 파일 읽기 금지
- example/sample env 파일은 읽어도 된다는 예외 규칙
- Extension → Supabase → Dashboard 데이터 계약을 한쪽만 깨는 변경 금지
- 하위 `CLAUDE.md`가 루트보다 우선한다는 계층 규칙
- 사용자가 명시하지 않은 삭제, 초기화, 대량 포맷팅, 설정 덮어쓰기 금지

제약 조건에 넣지 말아야 할 것:
- graphify를 먼저 읽는 절차 (→ 작업 흐름에 둔다)
- build/test 명령어 (→ 도구에 둔다)
- 일반적인 코딩 취향 (→ 개발 원칙 또는 코딩 컨벤션에 둔다)
- 생성물/외부 의존성 제외 규칙 (→ 작업 흐름에 둔다)

## 프로젝트 소개 작성 규칙
- 2-4문장으로 제한한다.
- 프로젝트가 무엇인지, 누구를 위한 것인지, 무엇을 절대 하지 않는지만 설명한다.
- 세부 기능 목록이나 긴 배경 설명은 넣지 않는다.

하위 폴더에서는 제목을 `## 이 폴더의 역할`로 바꿔도 된다.
이 경우 "이 폴더가 전체 시스템에서 어떤 책임을 갖는지"만 1-3문장으로 쓴다.

## 디렉터리 지도 작성 규칙
- 루트에서 중요한 폴더와 핵심 파일만 보여준다.
- 각 항목은 "어디에 무엇이 있는지"를 판단할 수 있을 정도로만 설명한다.
- `node_modules`, `.next`, `.venv`, cache 등 생성물은 지도에 넣지 않는다.
- 하위 폴더에 `CLAUDE.md`가 있으면 지도에 표시한다.

하위 폴더의 디렉터리 지도는 해당 폴더 안의 핵심 파일만 보여준다.
루트 전체 구조를 반복하지 않는다.

## 작업 흐름 작성 규칙
- 에이전트가 바로 코딩하지 않도록 읽기 → 계획 → 구현 → 검증 순서를 명시한다.
- graphify 읽기 규칙은 여기에 둔다.
- 하위 폴더 작업 전 해당 폴더의 `CLAUDE.md`를 읽는 절차를 포함한다.
- 코드 파일 변경 후 graphify rebuild 절차를 포함한다.

하위 폴더의 작업 흐름은 더 구체적으로 쓴다.
예: dashboard는 관련 page/component/query를 먼저 읽고, supabase는 schema/RLS/trigger 영향을 먼저 확인한다.

## 도구 작성 규칙
- 실제로 이 저장소에서 쓰는 명령만 적는다.
- 명령은 복사해서 실행 가능한 형태로 쓴다.
- 세부 폴더 전용 명령은 가능하면 하위 `CLAUDE.md`에 둔다.
- 루트에는 공통 검색, 대시보드 build/lint, 설정 동기화, graphify 갱신 정도만 둔다.

하위 폴더의 도구 섹션에는 그 폴더에서 실제로 실행할 명령만 둔다.
명령이 로컬 env나 비밀값을 읽는다면 "사용자가 명시적으로 요청한 경우" 같은 조건을 함께 적는다.

## 아키텍처 작성 규칙
- 계층 간 책임만 요약한다.
- 루트에서는 Extension, Supabase, Dashboard, Docs/Scripts 정도의 큰 구조만 설명한다.
- 함수별 세부 설명은 하위 `CLAUDE.md` 또는 코드 근처 문서로 보낸다.

하위 폴더에서는 `아키텍처` 대신 `디렉터리 지도`와 `폴더별 규칙`에 흡수해도 된다.
반복 설명보다 "수정할 때 어떤 파일 계약을 함께 봐야 하는지"를 우선한다.

## 도메인 컨텍스트 작성 규칙
- 프로젝트 판단에 필요한 핵심 용어만 둔다.
- 예: AX 전환 지표, 구조적 신호, minimal raw event, DB pre-aggregation, Asia/Seoul month key.
- 제품 철학 전체를 옮겨 쓰지 않는다.

하위 폴더에서는 해당 폴더가 다루는 도메인 용어만 둔다.
예: Extension은 payload 분류 신호, Supabase는 summary table, Dashboard는 activity label과 month key.

## 개발 원칙 작성 규칙
개발 원칙은 "어떻게 접근할 것인가"를 정의한다. 기술적 규칙(무엇을 쓸 것인가)은 코딩 컨벤션에 둔다.

포함할 것:
- 대원칙 한 줄 슬로건 (예: "Karpathy-first, TDD-when-risky")
- 코드를 건드리기 전 요구사항·가정·성공 기준을 명확히 한다는 원칙
- TDD 적용 기준: 버그 수정·비즈니스 로직·리팩토링·회귀 위험 → 적용 / UI 스타일·문구·로그·프로토타입 → 선택적
- 완료 기준: TDD 대상이면 테스트 통과, 그 외면 lint 통과 + 동작 직접 확인
- TDD 세부 규칙은 스킬 경로(`~/.claude/skills/tdd/`)로 참조만 한다 — 내용을 여기에 복사하지 않는다

넣지 말아야 할 것:
- TDD 안티패턴 목록 전문 (→ 스킬 파일에 있다)
- Red-Green-Refactor 사이클 상세 설명 (→ 스킬 파일에 있다)
- 언어/프레임워크별 테스트 작성법

## 코딩 컨벤션 작성 규칙
코딩 컨벤션은 "이 프로젝트의 기술적 규칙"을 정의한다. 접근 방식(사고법, 변경 범위)은 개발 원칙에 둔다.

- 모든 폴더에 공통 적용되는 규칙만 둔다.
- 언어/프레임워크별 세부 규칙은 하위 `CLAUDE.md`로 내린다.
- "기존 패턴 우선", "작은 변경 선호", "계층 계약 함께 검토"처럼 넓은 규칙만 유지한다.
- 개발 원칙과 겹치는 내용(surgical changes 등)은 한쪽에만 둔다.

하위 폴더에서는 제목을 `## 폴더별 규칙`으로 써도 된다.
이 섹션에는 framework/API/DB/문서 작성 등 해당 폴더에서만 중요한 기술 규칙을 둔다.

## 하위 폴더별 작성 방향
- `dashboard/CLAUDE.md`: Next.js App Router, Supabase SSR, server/client component 경계, query/component/page 수정 흐름을 중심으로 쓴다.
- `chatgpt-usage-tracker/CLAUDE.md`: Manifest V3, `background.js`, `supabase.js`, payload privacy, sync queue, 수동 브라우저 검증을 중심으로 쓴다.
- `supabase/CLAUDE.md`: `initial_setup.sql`, table, RLS, trigger, summary table, schema 변경 검증을 중심으로 쓴다.
- `docs/CLAUDE.md`: 문서와 코드 불일치 방지, 오래된 설계 표시, privacy-safe 예시 사용을 중심으로 쓴다.
- `docs/HARNESS/CLAUDE.md`: 운영 선택지, 저장 전략, 권한/정책, 실패 루프를 중심으로 쓴다.
- `scripts/CLAUDE.md`: 로컬 자동화, env 접근 주의, idempotent script 작성 기준을 중심으로 쓴다.

## 길이 제한
- 모든 `CLAUDE.md`는 300줄 이하를 반드시 유지한다.
- 루트 목표 길이는 80-120줄이다.
- 하위 폴더 목표 길이는 40-90줄이다.
- 내용이 길어지면 하위 `CLAUDE.md`나 별도 문서로 분리한다.

## 제외할 내용
- 긴 프로젝트 문서 링크 목록
- README 수준의 상세 설치 안내
- 하위 폴더 내부 구현 세부사항
- 스킬 파일에 이미 있는 내용의 복사본 (TDD 사이클, anti-pattern 목록 등)
- 이미 `.gitignore`나 도구 설정으로 충분히 관리되는 일반 설명
- 현재 코드와 맞지 않는 오래된 규칙
